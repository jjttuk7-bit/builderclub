# 빌더클럽 Builders Club PRD

## 0. 문서 개요

### 제품명
**빌더클럽 Builders Club**

### 제품 한 줄 설명
빌더클럽은 각자의 지식과 노하우를 공유하고, 각자가 만드는 서비스나 프로그램의 제작 과정을 기록하는 커뮤니티형 대시보드 플랫폼이다.

### 제품 목적
빌더클럽의 1차 목적은 **지식과 노하우 공유**이다.  
2차 목적은 **각자의 프로젝트 제작 과정을 기록하고, 진척을 공유하며, 서로 피드백을 주고받는 것**이다.

즉, 빌더클럽은 완성작을 자랑하는 공간이 아니라,  
**배운 것, 막힌 것, 해결한 것, 만들고 있는 과정을 함께 축적하는 공간**이다.

---

# 1. 문제 정의

## 1.1 현재 문제

서비스나 프로그램을 만들 때 사람들은 다음과 같은 문제를 겪는다.

1. 혼자 만들다 보니 중간에 막히면 쉽게 멈춘다.
2. 에러 해결 과정이나 시행착오가 개인 메모에 흩어져 사라진다.
3. Claude Code, Codex, ChatGPT 같은 AI 도구를 어떻게 활용했는지 기록되지 않는다.
4. 좋은 노하우가 있어도 공유할 공간이 부족하다.
5. 완성된 결과물만 공유되기 때문에 실제 제작 과정에서 배운 내용은 축적되지 않는다.
6. 프로젝트를 진행하면서 스스로 어디까지 왔는지 확인하기 어렵다.
7. 다른 사람의 서비스를 잠재 사용자 입장에서 피드백해주는 구조가 부족하다.

## 1.2 해결하고자 하는 핵심 문제

빌더클럽은 다음 문제를 해결한다.

> “각자가 만들면서 배운 지식과 노하우를 공유하고, 프로젝트 제작 과정을 기록으로 남겨 함께 성장할 수 있는 공간이 필요하다.”

# 2. 제품 비전과 핵심 방향

## 2.1 제품 비전

빌더클럽은 매달 각자가 하나의 서비스나 프로그램을 만들고,  
그 과정에서 생긴 지식, 노하우, 에러 해결, AI 프롬프트, 기획 변경, 사용자 피드백을 기록하는 커뮤니티 플랫폼이다.

## 2.2 핵심 방향

빌더클럽의 방향은 다음과 같다.

1. **지식 공유 중심**
   - 개발 팁, 기획 노하우, 디자인 인사이트, 배포 방법, AI 활용법 등을 공유한다.

2. **프로젝트 기록 보조**
   - 각자의 프로젝트를 선언하고, 빌드로그를 작성하며, 진행률을 기록한다.

3. **과정 중심 커뮤니티**
   - 완성작보다 만드는 과정, 시행착오, 해결 과정에 더 높은 가치를 둔다.

4. **사용자 관점 피드백**
   - 서로의 서비스를 잠재 사용자 입장에서 보고 의견을 남긴다.

5. **월간 빌드 리듬**
   - 한 달 단위로 만들고, 기록하고, 공유하는 흐름을 만든다.

# 3. 타깃 사용자

## 3.1 주요 사용자

### 1) 사이드 프로젝트를 만드는 사람
- 개인 서비스, 웹앱, 자동화 도구, 포트폴리오, AI 서비스 등을 만드는 사람
- 혼자 만들지만 중간중간 피드백과 동기부여가 필요한 사람

### 2) AI 코딩 도구를 활용하는 사람
- Claude Code, Codex, Cursor, ChatGPT 등을 활용해 개발하는 사람
- 좋은 프롬프트나 해결 과정을 기록하고 싶은 사람

### 3) 개발/기획/디자인 노하우를 공유하고 싶은 사람
- 자신이 알게 된 지식을 정리하고 공유하고 싶은 사람
- 다른 사람의 시행착오에서 배우고 싶은 사람

### 4) 프로젝트 기록을 남기고 싶은 학습자
- AI, 개발, 데이터 분석, 디자인 등을 배우는 사람
- 프로젝트 기반으로 성장 기록을 남기고 싶은 사람

## 3.2 사용자 니즈

| 사용자 니즈 | 설명 |
|---|---|
| 지식 공유 | 새로 배운 개발, 기획, 디자인, AI 활용 노하우를 공유하고 싶다. |
| 질문 해결 | 막힌 문제를 질문하고 답변을 받고 싶다. |
| 제작 기록 | 내가 무엇을 만들었고 어떻게 해결했는지 남기고 싶다. |
| 진척 확인 | 현재 프로젝트 진행 상황을 스스로 확인하고 싶다. |
| 피드백 | 내가 만든 서비스가 사용자 입장에서 이해되는지 의견을 받고 싶다. |
| 아카이브 | 좋은 글, 프롬프트, 해결 방법을 나중에 다시 찾고 싶다. |

# 10. 성공 지표와 운영 기준

## 10.1 핵심 성공 지표

빌더클럽은 상업적 전환보다 커뮤니티 활동성과 지식 축적을 우선으로 본다.

| 지표 | 의미 |
|---|---|
| 주간 지식 공유 글 수 | 회원들이 얼마나 노하우를 공유하는지 |
| 질문 답변률 | 질문이 방치되지 않고 해결되는지 |
| 빌드로그 작성 수 | 프로젝트 제작 과정이 기록되는지 |
| 프로젝트 등록 수 | 실제 만들기 활동이 발생하는지 |
| 피드백 댓글 수 | 서로 사용자 관점 의견을 주는지 |
| 재방문율 | 커뮤니티가 지속적으로 사용되는지 |
| 검색 이용 수 | 축적된 지식이 다시 활용되는지 |

## 10.2 초기 목표

초기 1개월 기준 목표는 다음과 같다.

| 항목 | 목표 |
|---|---:|
| 가입 회원 | 30명 |
| 등록 프로젝트 | 10개 |
| 지식 공유 글 | 30개 |
| 질문 글 | 20개 |
| 답변률 | 70% 이상 |
| 빌드로그 | 50개 |
| 피드백 요청 | 10개 |
| 완성작 공유 | 3개 이상 |

## 10.3 운영 원칙

1. 완성작보다 과정을 존중한다.
2. 질문을 부끄럽게 만들지 않는다.
3. 피드백은 비난이 아니라 개선 중심으로 한다.
4. AI 프롬프트와 시행착오도 중요한 기록으로 본다.
5. 좋은 글은 자료실로 옮겨 재활용한다.
6. 매달 마지막 주에는 완성작과 회고를 공유한다.

## 10.4 커뮤니티 글쓰기 원칙

### 좋은 글의 기준
- 내가 겪은 상황이 구체적으로 적혀 있다.
- 무엇을 시도했는지 적혀 있다.
- 실패한 방법도 함께 적혀 있다.
- 해결 방법이 다른 사람에게 재사용 가능하다.
- 사용한 프롬프트나 코드가 포함되어 있다.
- 다음에 같은 문제를 겪는 사람이 참고할 수 있다.

### 피드백 원칙
```text
좋았던 점 → 헷갈린 점 → 개선 제안
```

예시:

```text
좋았던 점:
첫 화면에서 서비스가 지식 공유 커뮤니티라는 점은 잘 느껴졌습니다.

헷갈린 점:
프로젝트 기록이 어디에서 시작되는지 조금 찾기 어려웠습니다.

개선 제안:
대시보드 상단에 ‘내 프로젝트 등록하기’ 버튼을 작게 추가하면 좋겠습니다.
```

# 14. 최종 요약

빌더클럽은 단순한 게시판 사이트가 아니다.

빌더클럽의 1차 목적은 **각자의 지식과 노하우를 공유하는 것**이다.  
2차 목적은 **각자의 프로젝트를 기록하고, 진척을 확인하며, 피드백을 주고받는 것**이다.

따라서 초기 대시보드와 기능 설계는 다음 우선순위를 따른다.

```text
1순위: 지식 공유
2순위: 질문과 노하우
3순위: 빌드로그
4순위: 프로젝트 기록
5순위: 피드백
6순위: 완성작 공유
```

이 구조로 만들면 빌더클럽은 상업적인 SaaS 대시보드가 아니라,  
**함께 배우고, 함께 기록하고, 함께 성장하는 빌더 커뮤니티**가 된다.
