# 빌더클럽 실제 앱 구현 계획

작성일: 2026-05-01

## 1. 목적

이 문서는 현재 정적 HTML 프로토타입을 실제 저장, 조회, 작성, 댓글, 공개 범위가 동작하는 앱으로 전환하기 위한 실행 계획이다.

지금까지 만든 산출물은 다음 역할을 한다.

| 문서/파일 | 역할 |
|---|---|
| `dashboard.html` 외 정적 HTML | 화면 구조 프로토타입 |
| `docs/builderclub-progress-and-next-steps.md` | 지금까지의 작업 기록 |
| `docs/builderclub-data-model.md` | 저장해야 할 데이터 구조 |
| `docs/builderclub-app-routes.md` | 실제 앱 URL과 화면 흐름 |
| `docs/builderclub-component-map.md` | 재사용할 UI 컴포넌트 기준 |

이 문서는 위 산출물을 바탕으로 실제 개발 순서를 정리한다.

## 2. MVP 목표

빌더클럽 MVP는 아래 흐름이 실제로 동작하는 것을 목표로 한다.

```text
빌더가 자기 빌더룸에 들어간다
→ 프로젝트를 확인한다
→ 빌드로그를 작성한다
→ 막힌 내용은 질문으로 전환한다
→ 만든 화면은 피드백을 요청한다
→ 해결한 내용은 지식 공유로 남긴다
→ 대시보드에서 전체 활동을 본다
```

MVP에서 반드시 필요한 기능:

- 빌더 프로필과 빌더룸 조회
- 프로젝트 목록과 프로젝트 상태 조회
- 빌드로그 작성, 목록, 상세
- 질문 작성, 목록, 상세, 댓글
- 피드백 요청 작성, 목록, 상세, 댓글
- 지식 공유 작성, 목록, 상세
- 대시보드 활동 피드
- 태그와 공개 범위

MVP 이후로 미룰 수 있는 기능:

- 알림
- 북마크
- 고급 검색
- 외부 공개 페이지
- 프로젝트 공동 작업자 권한
- 파일 업로드 고도화
- 관리자 화면

## 3. 권장 기술 방향

정적 HTML을 실제 앱으로 전환할 때는 아래 방향이 자연스럽다.

| 영역 | 권장 방향 | 이유 |
|---|---|---|
| 프론트엔드 | React 또는 Next.js | 현재 화면 구조를 컴포넌트로 옮기기 쉽다 |
| 스타일 | CSS Modules, Tailwind, 또는 기존 CSS 토큰 이식 | 프로토타입의 색상과 레이아웃을 유지하기 쉽다 |
| 데이터베이스 | PostgreSQL 계열 | 문서화한 관계형 데이터 모델과 잘 맞는다 |
| 백엔드/API | Next.js 서버 액션/API 라우트 또는 별도 API 서버 | 작성/조회/댓글 흐름을 단계적으로 붙이기 쉽다 |
| 인증 | 이메일/소셜 로그인 | 빌더별 개인공간과 공개 범위 제어에 필요하다 |
| 파일 저장 | 객체 스토리지 | 피드백 이미지, 첨부 자료 저장에 필요하다 |

초기에는 기술을 과하게 늘리지 않는 것이 좋다. 먼저 하나의 앱 안에서 화면, 데이터, 인증, 작성 흐름을 붙이고, 이후 필요할 때 분리한다.

## 4. 구현 단계 요약

```text
1단계: 앱 뼈대 만들기
2단계: 데이터 모델과 샘플 데이터 만들기
3단계: 읽기 화면 연결하기
4단계: 작성 화면 저장 연결하기
5단계: 댓글과 전환 흐름 만들기
6단계: 대시보드 활동 피드 만들기
7단계: 인증과 공개 범위 보강하기
8단계: MVP 검증과 배포 준비하기
```

## 5. 1단계: 앱 뼈대 만들기

목표:

- 정적 HTML의 핵심 화면을 실제 앱 라우트로 옮긴다.
- 아직 데이터 저장은 붙이지 않고, 샘플 데이터로 화면 구조를 먼저 고정한다.

작업:

1. 앱 프로젝트를 만든다.
2. 공통 레이아웃 `AppShell`을 만든다.
3. `SidebarNav`, `TopBar`, `PageHeader`를 만든다.
4. 라우트를 만든다.

필수 라우트:

```text
/dashboard
/builders/[handle]
/projects
/logs
/logs/new
/logs/[id]
/questions
/questions/new
/questions/[id]
/feedback
/feedback/new
/feedback/[id]
/knowledge
/knowledge/new
/knowledge/[id]
```

완료 기준:

- 모든 라우트가 열려야 한다.
- 각 라우트의 제목과 주요 액션이 보여야 한다.
- 기존 `index.html`은 삭제하지 않는다.

테스트 기준:

- 라우트별 렌더링 테스트
- 사이드바 메뉴 링크 테스트
- 현재 페이지 표시 테스트

## 6. 2단계: 데이터 모델과 샘플 데이터 만들기

목표:

- `docs/builderclub-data-model.md` 기준으로 실제 저장 구조를 만든다.
- 개발 중 화면 확인을 위한 샘플 데이터를 준비한다.

우선 테이블:

```text
builders
projects
build_logs
questions
feedback_requests
knowledge_posts
comments
tags
taggings
activity_feed
attachments
```

작업:

1. 테이블 스키마를 작성한다.
2. 상태값과 공개 범위 enum을 정의한다.
3. 빌더 A, 빌더 B, 빌더 C 샘플 데이터를 만든다.
4. 각 빌더에 프로젝트와 빌드로그 샘플을 연결한다.
5. 질문, 피드백, 지식 공유 샘플을 만든다.

완료 기준:

- 샘플 데이터만으로 대시보드, 빌더룸, 목록, 상세 화면이 채워진다.
- 모든 주요 콘텐츠가 `builder_id` 또는 작성자 정보를 가진다.
- 공개 범위가 없는 주요 콘텐츠가 없어야 한다.

테스트 기준:

- 스키마 생성 테스트
- 샘플 데이터 조회 테스트
- 관계 연결 테스트

## 7. 3단계: 읽기 화면 연결하기

목표:

- 정적 텍스트를 실제 데이터 조회 결과로 바꾼다.

작업 순서:

1. `/dashboard`에 `activity_feed`를 연결한다.
2. `/builders/[handle]`에 빌더, 프로젝트, 최근 로그를 연결한다.
3. `/projects`에 프로젝트 목록을 연결한다.
4. `/logs`, `/questions`, `/feedback`, `/knowledge` 목록을 연결한다.
5. 각 상세 화면을 연결한다.

완료 기준:

- 목록에서 상세로 이동할 수 있다.
- 상세에서 작성자와 프로젝트 정보가 함께 보인다.
- 비어 있는 데이터 상태가 깨지지 않는다.

테스트 기준:

- 목록 렌더링 테스트
- 상세 조회 테스트
- 없는 ID 접근 시 빈 상태 또는 404 처리 테스트

## 8. 4단계: 작성 화면 저장 연결하기

목표:

- 작성 화면에서 입력한 내용을 실제 데이터로 저장한다.

작업 순서:

1. `/logs/new` 저장을 연결한다.
2. `/questions/new` 저장을 연결한다.
3. `/feedback/new` 저장을 연결한다.
4. `/knowledge/new` 저장을 연결한다.
5. 저장 후 상세 화면으로 이동한다.

공통 저장 규칙:

- 제목 또는 핵심 내용은 필수다.
- 작성자는 현재 빌더로 연결한다.
- 공개 범위 기본값은 `club`으로 둔다.
- 저장 성공 시 `activity_feed`에 기록한다.
- 저장 실패 시 입력값을 잃지 않는다.

완료 기준:

- 작성한 콘텐츠가 목록과 상세에 바로 보인다.
- 필수값이 없으면 저장되지 않는다.
- 저장 후 이동 경로가 일관된다.

테스트 기준:

- 폼 검증 테스트
- 저장 성공 테스트
- 저장 실패 테스트
- 저장 후 리다이렉트 테스트

## 9. 5단계: 댓글과 전환 흐름 만들기

목표:

- 질문, 피드백, 지식 상세에서 대화가 쌓이게 한다.
- 빌드로그와 질문을 다른 콘텐츠로 전환할 수 있게 한다.

댓글 우선 적용 화면:

```text
/questions/[id]
/feedback/[id]
/knowledge/[id]
/logs/[id]
```

전환 흐름:

```text
build_logs → questions
build_logs → feedback_requests
build_logs → knowledge_posts
questions → knowledge_posts
feedback_requests → build_logs
```

완료 기준:

- 질문 상세에서 답변 댓글을 남길 수 있다.
- 피드백 상세에서 피드백 댓글을 남길 수 있다.
- 빌드로그 상세에서 질문 또는 지식 공유 작성으로 일부 내용이 전달된다.
- 질문이 해결되면 지식 공유 초안으로 전환할 수 있다.

테스트 기준:

- 댓글 작성 테스트
- 채택 답변 표시 테스트
- 전환 시 source ID 유지 테스트
- 전환 후 원본 상세에서 연결 링크 표시 테스트

## 10. 6단계: 대시보드 활동 피드 만들기

목표:

- 전체 커뮤니티 활동 중심 대시보드를 실제 데이터로 만든다.

피드에 포함할 활동:

- 새 빌드로그 작성
- 새 질문 등록
- 질문 답변 작성
- 피드백 요청 등록
- 피드백 댓글 작성
- 지식 공유 발행
- 프로젝트 상태 변경

완료 기준:

- 각 작성/댓글/상태 변경 후 대시보드에 활동이 쌓인다.
- 활동 피드에서 원본 콘텐츠로 이동할 수 있다.
- 빌더, 프로젝트, 콘텐츠 타입이 표시된다.

테스트 기준:

- 활동 생성 테스트
- 최신순 정렬 테스트
- 콘텐츠 타입별 링크 테스트

## 11. 7단계: 인증과 공개 범위 보강하기

목표:

- 빌더별 개인공간과 공개 범위를 실제 권한으로 연결한다.

작업:

1. 로그인 사용자를 `builders.user_id`와 연결한다.
2. `/me`를 내 빌더룸으로 연결한다.
3. `private`, `club`, `public`, `archived` 공개 범위를 적용한다.
4. 작성자만 수정 가능한 규칙을 만든다.

완료 기준:

- 내 비공개 글은 나만 볼 수 있다.
- 클럽 공개 글은 로그인한 멤버가 볼 수 있다.
- 작성자가 아닌 사용자는 수정 버튼을 볼 수 없다.
- 비로그인 사용자의 접근 정책이 명확하다.

테스트 기준:

- 작성자 권한 테스트
- 비공개 콘텐츠 접근 제한 테스트
- 클럽 공개 콘텐츠 접근 테스트

## 12. 8단계: MVP 검증과 배포 준비

목표:

- 실제 사용자가 빌더클럽의 핵심 루프를 한 번 완주할 수 있게 만든다.

검증 시나리오:

1. 새 빌더로 로그인한다.
2. 내 빌더룸에 들어간다.
3. 프로젝트를 확인한다.
4. 빌드로그를 작성한다.
5. 빌드로그에서 질문을 만든다.
6. 질문에 댓글을 남긴다.
7. 해결 내용을 지식 공유로 전환한다.
8. 대시보드에서 모든 활동을 확인한다.

완료 기준:

- 위 시나리오를 끊김 없이 수행할 수 있다.
- 모바일과 데스크톱에서 핵심 화면이 깨지지 않는다.
- 빈 상태, 오류 상태, 로딩 상태가 존재한다.
- 최소 테스트가 통과한다.

## 13. 정적 HTML에서 재사용할 것

재사용할 것:

- 정보 구조
- 화면 흐름
- 필수 입력 항목
- 대시보드의 커뮤니티 활동 중심 구성
- 빌더룸의 개인 작업 공간 개념
- 목록과 상세의 콘텐츠 분류
- 색상 토큰과 카드 밀도

다시 설계할 것:

- 중복 CSS
- 하드코딩된 예시 텍스트
- HTML 파일 단위 링크
- 정적 상태값
- 저장되지 않는 폼

## 14. 첫 번째 실제 개발 스프린트 제안

가장 먼저 할 스프린트는 아래 범위가 좋다.

스프린트 목표:

```text
샘플 데이터 기반으로 실제 앱 라우트와 읽기 화면을 만든다.
```

포함 범위:

- 앱 프로젝트 생성
- 공통 레이아웃
- 대시보드
- 빌더룸
- 프로젝트 목록
- 빌드로그 목록/상세
- 샘플 데이터

제외 범위:

- 로그인
- 실제 DB 저장
- 댓글 저장
- 파일 업로드
- 알림

이유:

- 화면 구조와 데이터 형태를 먼저 고정할 수 있다.
- 이후 작성/저장 기능을 붙일 때 되돌림이 적다.
- 현재 정적 HTML의 가치가 가장 빠르게 실제 앱 구조로 옮겨진다.

## 15. 다음 작업

이제 선택지는 두 가지다.

1. 실제 앱 프로젝트를 새로 만든다.
2. 앱을 만들기 전에 더 구체적인 `1단계 앱 뼈대 구현 계획`을 `docs/plans/` 아래에 작성한다.

권장 순서는 2번이다. 먼저 작은 실행 계획을 만든 뒤, 그 계획대로 앱 프로젝트를 생성하면 작업 단위가 훨씬 선명해진다.
